# Context Mesh Framework Reference

> **For AI Agents**: This file contains the framework rules and patterns. Load this file when working with Context Mesh projects.

---

## Framework Overview

**Context Mesh** is a 3-step framework for AI-First development:

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   INTENT    │ ──► │    BUILD    │ ──► │    LEARN    │
│  What & Why │     │  AI + Human │     │   Update    │
└─────────────┘     └─────────────┘     └─────────────┘
```

1. **Intent** - Define what to build and why (create context)
2. **Build** - AI generates code following context (human supervises)
3. **Learn** - Update context with outcomes (refine and improve)

**Key Principle**: Context is primary. Code is its manifestation.

Context Mesh structures AI-assisted development around **explicit context artifacts**, governed by humans and executed by agents.

Phases are sequential, but not automatic. Transitions require explicit human intent.

## File Type Separation Rules (CRITICAL)

### Feature Files (`context/intent/feature-*.md`)

**MUST contain:**
- **What** - Description of what the feature does
- **Why** - Business/user value, problem it solves
- **Acceptance Criteria** - Functional requirements (what must work)
- **Related** - Links to project intent and decisions
- **Status** - Current status (Draft, In Progress, Completed, etc.)

**MUST NOT contain:**
- ❌ Technical implementation details (→ `decisions/`)
- ❌ Code patterns or examples (→ `knowledge/patterns/`)
- ❌ Step-by-step execution instructions (→ `agents/`)
- ❌ Definition of Done (→ `agents/` or feature DoD section if needed)

**Example structure:**
```markdown
# Intent: Feature - [Name]

## What
[What this feature does]

## Why
[Why we need it - business/user value]

## Acceptance Criteria
- [ ] Criterion 1
- [ ] Criterion 2

## Related
- [Project Intent](project-intent.md)
- [Decision: [Name]](../decisions/[number]-[name].md)

## Status
- **Created**: YYYY-MM-DD (Phase: Intent)
- **Status**: Draft
```

---

### Decision Files (`context/decisions/*.md`)

**MUST contain:**
- **Context** - Situation, requirements, constraints
- **Decision** - Technical choice/approach selected
- **Rationale** - Why this approach was chosen
- **Alternatives Considered** - Other options evaluated and why not chosen
- **Outcomes** - What happened after implementation (updated in Learn step)
- **Related** - Links to features, project intent, other decisions
- **Status** - Current status (Proposed, Accepted, Superseded, etc.)

**MUST NOT contain:**
- ❌ Feature requirements (→ `intent/feature-*.md`)
- ❌ Code examples or patterns (→ `knowledge/patterns/`)
- ❌ Execution steps (→ `agents/`)

**Key Principle**: Decisions are immutable once accepted. They are evolved through supersession, not rewriting.

**Example structure:**
```markdown
# Decision: [Name]

## Context
[Situation and requirements]

## Decision
[Technical approach chosen]

## Rationale
[Why this approach]

## Alternatives Considered
- [Alternative 1] - [Why not chosen]
- [Alternative 2] - [Why not chosen]

## Outcomes
[Updated after implementation]

## Related
- [Feature: [Name]](../intent/feature-[name].md)
- [Project Intent](../intent/project-intent.md)

## Status
- **Created**: YYYY-MM-DD (Phase: Intent)
- **Status**: Accepted
```

---

### Knowledge Files (`context/knowledge/patterns/*.md` and `anti-patterns/*.md`)

**MUST contain:**
- **Description** - What the pattern/anti-pattern is
- **When to Use** - Specific scenarios (for patterns)
- **Why it's problematic** - For anti-patterns
- **Examples** - Code examples showing the pattern
- **Related** - Links to decisions that use this pattern
- **Status** - Current status (Active, Deprecated, Superseded)

**MUST NOT contain:**
- ❌ Feature requirements (→ `intent/`)
- ❌ Technical decisions (→ `decisions/`)
- ❌ Execution steps (→ `agents/`)

---

### Agent Files (`context/agents/agent-*.md`)

**MUST contain:**
- **Purpose** - What this agent does
- **Context Files to Load** - Which context files to reference
- **Execution Steps** - Step-by-step instructions
- **Definition of Done** - Technical criteria for completion
- **Scope** - What this agent can/cannot do

**MUST NOT contain:**
- ❌ Feature requirements (→ `intent/`)
- ❌ Technical decisions (→ `decisions/`)
- ❌ Code patterns (→ `knowledge/`)

**Key Principle**: Agents **reference** context, not duplicate it. Agents are **operators**, not authorities. They never hold authority or make decisions.

## Plan, Approve, Execute Pattern (MANDATORY)

**CRITICAL**: When implementing ANY feature, bug fix, or update, you MUST follow this pattern:

### 1. PLAN (Do this first - DO NOT SKIP)

**Before writing any code, you MUST:**

1. **Load framework context:**
   - Load this file (`@context/.context-mesh-framework.md`) if it exists
   - Understand Context Mesh structure and rules

2. **Load relevant context files:**
   - `@context/intent/project-intent.md` (always)
   - `@context/intent/feature-*.md` (for the feature being implemented)
   - `@context/decisions/*.md` (relevant decisions)
   - `@context/knowledge/patterns/*.md` (patterns to follow)
   - `@context/knowledge/anti-patterns/*.md` (what to avoid)

3. **Analyze existing codebase:**
   - Understand current structure
   - Identify existing code related to the feature
   - Check if feature already exists (for updates)

4. **Verify prerequisites:**
   - ✅ ADR (decision file) exists before implementation
   - ✅ Feature intent file exists
   - ✅ Patterns are understood

5. **Create and present the plan:**
   - Explain your implementation approach
   - List ALL files you will create
   - List ALL files you will modify
   - Show code structure/architecture
   - Explain how you'll follow decisions and patterns
   - Present the complete plan clearly
   - **CRITICAL**: Present the plan in conversation only - DO NOT create any files during planning
   - **Use context files** - Reference existing `@context/` files instead of creating temporary files

**Example plan presentation:**
```
PLAN for implementing [feature name]:

1. Context loaded:
   - @context/intent/feature-[name].md
   - @context/decisions/[number]-[name].md
   - @context/knowledge/patterns/[pattern].md

2. Approach:
   - [Explain technical approach based on decision]

3. Files to create:
   - [List files]

4. Files to modify:
   - [List files]

5. Structure:
   - [Show architecture/structure]
```

### 2. APPROVE (Wait for user approval - DO NOT SKIP)

**After presenting the plan:**

1. **Ask explicitly:**
   - "Should I proceed with this implementation plan?"
   - "Does this plan look correct?"

2. **DO NOT execute until user approves:**
   - ❌ Do not write any code
   - ❌ Do not create any files (plans exist only in conversation)
   - ❌ Do not modify anything
   - ❌ Do not save the plan as a file - use context files for reference instead

3. **If user requests changes:**
   - Update the plan
   - Present updated plan
   - Ask for approval again
   - Repeat until approved

### 3. EXECUTE (Only after approval)

**After user approves:**

1. **Implement according to approved plan:**
   - Follow the plan exactly as approved
   - Create only files from the approved plan
   - Modify only files from the approved plan
   - Follow all context files strictly
   - **CRITICAL**: If you create any file not in the approved plan, you MUST delete it immediately
   - **Use context files** - Reference `@context/` files instead of creating temporary documentation files

2. **Respect decisions and patterns:**
   - Follow technical decisions from `decisions/`
   - Use patterns from `knowledge/patterns/`
   - Avoid anti-patterns from `knowledge/anti-patterns/`

3. **Maintain code quality:**
   - Follow existing code style
   - Write clean, maintainable code
   - Add comments where needed

4. **Respect authority model:**
   - Agents are operators, not authorities
   - Never expand scope autonomously
   - Execution is always bounded by approved context

5. **File management and cleanup:**
   - **MANDATORY**: If you create any file not explicitly listed in the approved plan, you MUST delete it immediately
   - **Use context files** - Reference `@context/` files for documentation and reference instead of creating temporary files
   - Only files from the approved plan should remain after execution
   - Context files are the source of truth - do not duplicate information in temporary files

**CRITICAL RULES**: 
- Never skip Plan and Approve steps. Always explain first, wait for approval, then execute.
- Plans are ephemeral - they exist only in conversation. Do NOT create any files during planning.
- Use context files (`@context/`) for reference - do not create temporary files.
- After execution, document outcomes in `context/evolution/changelog.md`, not in separate files.
- **File cleanup**: If you create any file not explicitly in the approved plan, you MUST delete it immediately.

## When Creating Context Files

### When to CREATE files:

✅ **DO create when:**
- User explicitly requests (add feature, fix bug, etc.)
- User uses prompts like "add feature", "fix bug", "implement"
- User explicitly requests: "create", "make", "build", "generate"
- Following a Context Mesh prompt (add-feature.md, fix-bug.md, etc.)

### When to NOT create files:

❌ **DO NOT create when:**
- User asks a question (e.g., "How does X work?", "What is Y?")
- User asks for explanation or clarification
- User asks "What should I do?" or "What's the best approach?"
- User is exploring or learning about the codebase
- User asks for suggestions or recommendations
- **During PLAN phase** - Plans must be presented in conversation only, never saved as files. Use context files for reference instead.
- **During APPROVE phase** - No files should be created while waiting for approval

**When user asks questions:**
1. Answer using context from `@context/` files
2. DO NOT create any markdown files automatically
3. You CAN suggest updating Context Mesh if relevant, but:
   - Use format: "💡 **Suggestion**: Consider updating `@context/decisions/001-tech-stack.md` with this information..."
   - DO NOT create the file - just suggest it
   - Let the user decide if they want to update

## Updating Existing Features

When updating an existing feature (not creating new):

1. **Analyze existing code first:**
   - Understand what's already implemented
   - Identify existing files related to the feature

2. **Follow Plan, Approve, Execute:**
   - Plan what needs to change
   - Get approval
   - Make ONLY necessary modifications

3. **Make incremental changes only:**
   - Modify only what needs to change based on updated intent
   - Preserve existing code that doesn't need to change
   - Follow "Changes from Original" section if present in intent file

4. **Update related files:**
   - Update tests, documentation, and related code as needed

5. **Respect evolution rules:**
   - Context never mutates silently
   - History is immutable
   - Decisions are superseded, not rewritten
   - All evolution is recorded in `context/evolution/changelog.md`

## Definition of Done (DoD)

**DoD applies to Step 2 (Build) only** - when implementing features.

**Typical DoD Criteria:**
- [ ] Code implemented and working
- [ ] Tests passing (if applicable)
- [ ] Code follows patterns from `@context/knowledge/patterns/`
- [ ] Decisions respected from `@context/decisions/`
- [ ] Context updated after implementation (Context Mesh requirement)
- [ ] Acceptance criteria met (from feature intent)
- [ ] Scope matches approved plan (no silent scope expansion)

**DoD can be:**
- In `agent-*.md` files (for agent-based execution)
- In feature intent files (optional DoD section)
- Project-level (defined by team)

## Bidirectional Links

**CRITICAL**: Always create bidirectional links between related files:

- Feature files → Decision files
- Decision files → Feature files
- Features → Project Intent
- Decisions → Project Intent

**Link format:**
```markdown
## Related
- [Feature: User Auth](../intent/feature-user-auth.md)
- [Decision: Auth Approach](../decisions/002-auth-approach.md)
- [Project Intent](../intent/project-intent.md)
```

## Status Section

**Every context file MUST include a Status section:**

```markdown
## Status
- **Created**: YYYY-MM-DD (Phase: Intent/Build/Learn)
- **Status**: [Status Type]
- **Updated**: YYYY-MM-DD (Phase: Intent/Build/Learn) - [reason] (optional)
```

**Status Types:**
- **Intent Files**: Draft, In Progress, Completed, Deprecated, Resolved
- **Decision Files**: Proposed, Accepted, Superseded, Deprecated
- **Knowledge Files**: Active, Deprecated, Superseded

## Authority Model

### Humans
- Own intent
- Accept decisions
- Approve execution
- Validate learning

### MCP (Mesh Control Plane)
- Validates requests
- Enforces decisions and rules
- Orchestrates agents
- Produces context bundles

### Agents
- Analyze
- Propose
- Execute approved work
- Never decide or self-evolve

**Agents are operators, not authorities.**

All execution requests flow through MCP validation. Direct agent execution without MCP context bundling is invalid.

---

## Context Bundling Rules

- Context is loaded explicitly
- No implicit expansion
- Bundles are deterministic and immutable
- Only referenced artifacts are included
- Code is evidence, never intent

Agents operate only on provided bundles.

---

## Evolution Rules

- Context never mutates silently
- History is immutable
- Decisions are superseded, not rewritten
- Features may be completed, replaced, or abandoned
- Learning always references concrete outcomes
- Learning is explicit and human-validated
- Learning never runs automatically

All evolution is recorded in `context/evolution/changelog.md`.

---

## Prohibited Behaviors

The following are explicitly forbidden:

- Executing code without approved intent
- Modifying context without human validation
- Treating code as the source of truth
- Auto-learning or background adaptation
- Agent-driven decision making
- Implicit scope expansion
- Skipping Plan and Approve steps
- Creating files when user asks questions
- Creating any files during PLAN or APPROVE phases - plans must exist only in conversation
- Creating files not in the approved plan - any such files must be deleted immediately
- Creating temporary documentation files - use context files instead

Violating these rules breaks the framework contract.

1. ✅ **Separate concerns** - Features, decisions, knowledge, agents have different purposes
2. ✅ **Always Plan, Approve, Execute** - Never skip planning and approval

4. ✅ **Load context first** - Always load relevant context files before implementing
5. ✅ **Update context after** - Always update context in Learn step
6. ✅ **Bidirectional links** - Create links between related files
7. ✅ **Status tracking** - Include status in every context file
8. ✅ **Respect authority model** - Agents are operators, not authorities
9. ✅ **Context bundling** - Only operate on provided bundles
10. ✅ **Explicit evolution** - Never mutate context silently
11. ❌ **Don't mix concerns** - Don't put decision details in feature files
12. ❌ **Don't skip approval** - Always get user approval before executing
13. ❌ **Don't create files for questions** - Only create when explicitly requested
14. ❌ **Don't create files during planning** - Plans must be presented in conversation only, use context files for reference
15. ❌ **Don't create files outside approved plan** - Any file created not in the approved plan must be deleted immediately
