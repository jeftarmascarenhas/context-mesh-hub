# Context Mesh Framework — Project Contract

## Purpose

This document defines the **Context Mesh Framework contract** as applied in this repository.

It is the authoritative reference for:
- humans
- AI agents
- MCP orchestration

Any interaction with this repository MUST comply with the rules defined here.

This file is not a tutorial.
It is an **operational specification**.

---

## Core Principle

> **Context is Primary. Code is Manifestation.**

Code is never the source of truth.
Context is.

---

## Framework Overview

Context Mesh structures AI-assisted development around **explicit context artifacts**, governed by humans and executed by agents.

### Phases

The framework operates in three immutable macro phases:

1. **Intent**
2. **Build**
3. **Learn**

Phases are sequential, but not automatic.
Transitions require explicit human intent.

---

## Phase Definitions

### 1. Intent

**Intent defines WHAT and WHY.**

Intent artifacts:
- describe desired outcomes
- define scope and constraints
- include Acceptance Criteria
- reference applicable decisions

Intent artifacts:
- are written by humans
- may be drafted with AI assistance
- are authoritative once accepted

No code is written during the Intent phase.

---

### 2. Build

**Build defines HOW and executes change.**

Build operates under explicit governance:
- technical decisions
- patterns and anti-patterns
- approved execution scope

Build may include the following execution modes:
- Plan
- Approve
- Execute

Execution is always bounded by approved context.
Agents never expand scope autonomously.

---

### 3. Learn

**Learn captures outcomes and evolves context.**

Learn:
- is explicit
- is human-validated
- never runs automatically

Learning outputs may update:
- decisions
- knowledge (patterns / anti-patterns)
- evolution history

Learning never mutates context silently.

---

## Context Artifact Types

Context Mesh uses explicit artifact types:

### Intent
- Project Intent
- Feature Intent

### Decisions
- Technical and architectural choices
- Immutable once accepted
- Evolved through supersession, not rewriting

### Knowledge
- Patterns (what to follow)
- Anti-patterns (what to avoid)

### Agents
- Execution playbooks
- Contain Definition of Done
- Never hold authority

### Evolution
- Historical record of changes
- Explains what changed and why
- Preserves institutional memory

---

## Authority Model

### Humans
- Own intent
- Accept decisions
- Approve execution
- Validate learning

### MCP (Mesh Control Plane)
- Validates requests
- Enforces decisions and rules
- Orchestrates agents
- Produces context bundles

### Agents
- Analyze
- Propose
- Execute approved work
- Never decide or self-evolve

Agents are **operators**, not authorities.

---

## Context Bundling Rules

- Context is loaded explicitly
- No implicit expansion
- Bundles are deterministic and immutable
- Only referenced artifacts are included
- Code is evidence, never intent

Agents operate only on provided bundles.

---

## Evolution Rules

- Context never mutates silently
- History is immutable
- Decisions are superseded, not rewritten
- Features may be completed, replaced, or abandoned
- Learning always references concrete outcomes

All evolution is recorded in `context/evolution/changelog.md`.

---

## Prohibited Behaviors

The following are explicitly forbidden:

- Executing code without approved intent
- Modifying context without human validation
- Treating code as the source of truth
- Auto-learning or background adaptation
- Agent-driven decision making
- Implicit scope expansion

Violating these rules breaks the framework contract.

---

## Compliance Requirement

All contributors (human or AI):
- MUST load and respect this file
- MUST follow the Context Mesh workflow
- MUST update context after execution

Failure to comply results in:
- unreliable context
- unsafe execution
- degraded system behavior
