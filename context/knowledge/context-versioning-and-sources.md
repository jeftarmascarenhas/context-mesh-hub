# Context Mesh Hub: Where Templates and Framework Come From

## Where MCP Tools Get Templates (Prompt Packs)

MCP tools resolve **prompt templates** (e.g. `add-feature.md`, `clarify.md`) via `PromptResolver` in hub-core. Order:

1. **Repo override** (highest): `<repo_root>/.context-mesh/prompts/<template>.md`
2. **Pinned cached pack**: `~/.context-mesh-hub/prompt-packs/<pack>/<version>/<template>.md`  
   (version comes from `context/hub-manifest.json` in the project)
3. **Bundled fallback**: hub-core’s own `hub-core/prompt-packs/context-mesh-core/1.0.0/` (or 1.2.0 if pinned)

So templates are ultimately from **hub-core’s prompt-packs** (or overrides/cache). The project’s `context/hub-manifest.json` pins which pack/version is used.

## Where the Framework File Comes From

The file **`context/.context-mesh-framework.md`** (framework rules, Plan/Approve/Execute, quality gates) lives **inside the project repo**. It is:

- **Created** when the user runs `cm_init` or `cm_new_project` (content is generated by hub-core and written into the repo).
- **Not** fetched from a versioned URL on each run. There is no “latest framework” endpoint.
- **Not** versioned in a machine-readable way inside the file (no `version: 1.2.0` field).

So an old project (e.g. created with Context Mesh 1.1.0) has whatever framework content was generated at init time. Hub does not currently “upgrade” that file automatically.

## “Verify if context is using the last context mesh version and update”

- **Prompt pack (templates):**  
  - **Verify:** Yes. The user (or agent) can call `hub_prompts_status` to see current pack name and version (e.g. `context-mesh-core` 1.0.0).  
  - **Update:** Partially. The user/agent can call `hub_prompts_install("context-mesh-core", "1.2.0")` and then pin that version in the manifest. There is no “latest version” API; the agent or user must know that 1.2.0 exists (e.g. from docs or `cm_help`).
- **Framework file:**  
  - **Verify:** No. The framework file does not expose a version to the hub.  
  - **Update:** No. There is no tool that fetches “latest” framework content or overwrites `context/.context-mesh-framework.md`.

So the exact phrase “verify if context is using the last context mesh version and update the context” works **only for the prompt pack** (verify via `hub_prompts_status`, update by installing and pinning a known newer version). It does **not** yet verify or update the framework file to “the latest”.

## How the User Can Interact

1. **CLI (`cm`)**  
   - Run `cm` (no args) for the interactive menu, or `cm init`, `cm config`, `cm doctor`, `cm ui`, `cm setup-commands`, `cm agents`, `cm projects list|add|remove`. Slash commands are not in the CLI; they live in agent chat only.

2. **Agent chat (Cursor, and later Copilot, Claude CLI, Gemini CLI)**  
   - User runs `cm setup-commands` to install slash commands (e.g. `/cm-add-feature`, `/cm-build-plan`) into the agent command palette.  
   - User types in the agent chat: add a feature for login, or uses a slash command like `/cm-add-feature user login`.  
   - The **agent** (with MCP connected to hub-core) sees tool descriptions and can call e.g. `cm_add_feature`, `cm_new_project`, `hub_prompts_status`. Natural language or slash commands in agent chat drive the MCP tools.

   - User types in the agent chat or IDE: “start context new project” or “verify if context is using the last version…”.  
   - The **agent** (with MCP connected to hub-core) sees tool descriptions and can choose e.g. `cm_new_project` or `hub_prompts_status`. It works if the agent infers the right tool; the hub does not parse that exact sentence, the IDE’s model does.

**Summary:** The user can use **CLI**, **agent chat (slash commands from cm setup-commands)**, or **IDE chat**; they do not have to type slash commands in chat. For “start context new project”, agent chat (with MCP) can drive the right MCP tools.
